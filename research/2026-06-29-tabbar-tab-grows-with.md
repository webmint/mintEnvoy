# Research: TabBar tab grows with title instead of fixed max-width; long titles must truncate with ellipsis (nowrap+text-overflow), tab width stays stable, + button stays put — pure CSS/layout constraint, zero behaviour change, pixel-fidelity to design reference

**Date**: 2026-06-29
**Topic**: TabBar tab grows with title instead of fixed max-width; long titles must truncate with ellipsis (nowrap+text-overflow), tab width stays stable, + button stays put — pure CSS/layout constraint, zero behaviour change, pixel-fidelity to design reference
**Mode**: Enhancement
**Verdict**: Feasible

## Summary

The tab strip already has truncation machinery, but the width cap is bound to the wrong element. TabBar.css:88 caps the LABEL (.tabbar .tabs**tab-label) at max-width:200px, while the design's .tab cell — our .tabbar .tabs**tab-wrapper (Tabs.css:421) — carries no max-width and the design-fidelity-contract section 5 requires .tab max-width:220px. So each cell sizes to its content (method-chip + gap + label up to 200px + padding), grows with the title, varies with chip width, and exceeds the 220px target, pushing the + actions. The fix is pure CSS and .tabbar-scoped: add max-width:220px to the cell wrapper and remove the divergent label cap, reusing the base label ellipsis rule (Tabs.css:196) that already matches contract B1. deriveLabel returns a plain string, so no JS/markup/behaviour change is needed and existing tests stay green (the CT fidelity assertion that pins the old 200px label cap will need re-pointing to the 220px cell cap).

## Root Cause Hypothesis (WHY)

**Primary hypothesis**: The tab width cap is bound to the wrong element. The label is capped at 200px (TabBar.css:88) while the design's .tab cell (.tabbar .tabs\_\_tab-wrapper, Tabs.css:421) carries no max-width, so the cell grows with content and diverges from the contract's .tab max-width:220px. Relocating the cap to the cell (and removing the divergent label cap) fixes growth and aligns to fidelity; the base label ellipsis rule (Tabs.css:196) already truncates.

**Confidence**: Confirmed

## Approaches (HOW to change)

### Relocate the cap to the cell (contract-faithful)

- **Description**: Add max-width:220px to .tabbar .tabs**tab-wrapper (Tabs.css:421 — the design's .tab cell) and remove the divergent max-width:200px from .tabbar .tabs**tab-label (TabBar.css:88), keeping the base .tabs\_\_tab-label ellipsis rule. Pure CSS, .tabbar-scoped.
- **Addresses hypothesis**: A, B, C
- **Does NOT cover**: (none)
- **Pros**: Contract-faithful: caps the .tab cell at exactly 220px (design section 5); Reuses the existing base ellipsis rule (Tabs.css:196) — no new truncation code; Tab width is stable, independent of method-chip width; Scoped to .tabbar — bare .tabs consumers (section tabs, PrimitivesDemo) untouched; Pure CSS, zero behaviour/markup change; unit tests stay green; Resolves the growth under every framing — no flex-layer change needed
- **Cons**: Touches two files (Tabs.css add + TabBar.css removal); An existing CT fidelity assertion pinning the 200px label cap must be re-pointed to the 220px wrapper cap
- **Complexity**: Low

### Bump the label cap value (200 to 220 on the label)

- **Description**: Change .tabbar .tabs\_\_tab-label max-width from 200px to 220px in place (TabBar.css:88); leave the cap on the label.
- **Addresses hypothesis**: C
- **Does NOT cover**: A, B
- **Pros**: Single-line change, smallest possible diff; No new rule added to the wrapper
- **Cons**: Diverges from contract section 5 — max-width belongs on the .tab cell, not the label; Cell width stays = chip+label+padding, varies with chip and can still exceed 220px; A cell-level computed-style fidelity AC on the wrapper would still fail
- **Complexity**: Low

**Recommended approach**: Relocate the cap to the cell (contract-faithful) — Reuses the canonical truncation rule already present at src/renderer/src/components/molecules/Tabs.css:196 (it matches design-fidelity-contract B1) instead of writing any new styling, and derives its authority from design-fidelity-contract section 5. The single cell-level cap resolves the tab-growth symptom under every enumerated framing, so no flex-layer change is needed. It satisfies unchanged_behavior: the change is .tabbar-scoped and CSS-only, so select/close/reorder, the dirty dot, the method chip, active styling, and the rendered text stay byte-identical; the sole visible difference is how an overlong title is clipped. Bare .tabs consumers are untouched. Runtime behaviour is deferred to the computed-style probe captured under Open Uncertainties before finalizing.
