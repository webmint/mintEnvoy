## Summary

This plan implements a single-window app shell for the mintEnvoy renderer as a net-new `components/organisms/` tier (Shell + Titlebar + Sidebar + PaneSplit + Statusbar + hand-rolled Divider), backed by a net-new in-memory `settingsStore` that mirrors the existing `toastStore` zustand pattern and is the single source of truth for theme/accent/mstyle/sidebarWidth/paneRatio/sidebarCollapsed. The Shell applies `data-theme`/`data-accent`/`data-mstyle` to `<html>` via a store-driven effect (reusing the already-complete `tokens.css` contract; `data-accent` is set but visually inert this task) and drives sidebarWidth/paneRatio through rAF-batched CSS custom properties during drag, committing JS-clamped values to the store only on pointer release. No external dependency is added and no API/data-entity surface exists — the feature is renderer-only and in-memory.

## Technical Context

**Architecture**: Renderer process only. Presentation tier gains a new `organisms/` sublayer (below molecules); Support tier (`lib/`) gains `settingsStore`. Dependency direction preserved: organisms → molecules/lib; `lib/` never imports `components/`. App root (`App.tsx`) swaps the dev-only PrimitivesDemo child for `<Shell>` INSIDE the existing `<ToastProvider>` — the provider/viewport substrate is preserved, not replaced.
**Error Handling**: Renderer presentation code with no fallible I/O this task (in-memory store, DOM side-effects). The store→`<html>` effect overwrites attributes idempotently; window-resize/drag handlers clamp pure values. No try/catch surface beyond React's own; no swallowed errors.
**State Management**: One module-level zustand `settingsStore` (single instance, typed state + actions, plus a settingsStore-owned `reset()` action for tests). NB the reuse target is toastStore's _shape_, not a `reset()` action — toastStore exposes `clearAll()` and its tests wrap `getState().clearAll()` in a local `resetStore()` helper; settingsStore adds its own `reset()` action rather than mirroring a symbol toastStore does not have. State mutated only through actions (constitution §5). The in-drag CSS-var write is a transient render-only side-effect, NOT a state mutation — the authoritative value is committed via an action on `pointerup`.

## Constitution Compliance

- §2.1 (renderer-only, sandboxed): compliant — `settingsStore` stays in-memory, renderer-side, zero Node/electron imports (AC-15 grep gate). Persistence stays §6-deferred behind a future main-process IPC port.
- §2.3 (@renderer alias): compliant — all new organism↔molecule↔lib imports use `@renderer`, not deep relative paths.
- §3.1 (type safety, strict, no `any`): compliant — `SettingsState`, actions, and slot props fully typed; no per-file strict-weakening (AC-12 `typecheck:web`).
- §3.3 (naming): compliant — `settingsStore` (camelCase + `Store` suffix); `Shell`/`Titlebar`/`Sidebar`/`PaneSplit`/`Statusbar`/`Divider` (PascalCase, one-per-file).
- §5 / §4 (state via actions): compliant — drag/keyboard/Cmd-B/resize all commit through store actions; the rAF CSS-var write is render-only.
- §3.5 / §4 (styling): compliant — semantic class names bound to `tokens.css`, sibling `.css` files, no inline styles (AC-14), animations behind `prefers-reduced-motion`.
- §6.3 (search before building): compliant — reuses `toastStore` pattern, `cx()`, the Icon atom, and the `tokens.css` contract rather than re-defining them.
